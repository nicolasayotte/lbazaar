### F-18.4: Insufficient ADA for fee — NOT IMPLEMENTED

## F-16: Reward Invalidation on Refund | SHOULD — Partial (~70%)

### F-16.1: NFT certificate revoked on-chain and in platform — PARTIAL

**Platform-level (implemented):**
- `certificate_status` ENUM extended with `'revoked'` value (migration `2026_02_28_000003`)
- `RewardInvalidationService::invalidateRewards()` sets `certificate_status = 'revoked'` on refund
- Integration with `StripeService::refund()` and `CoursePurchaseService::refundPurchaseTransaction()` when `force=true`
- Student and admin in-app notifications written to `notifications` table
- Admin sees revocation status in ManageCourse roster (delivery_status = 'revoked')
- Student sees revoked reward in F-17 rewards view

**On-chain (NOT feasible with current architecture):**
- SC-E02-05 cannot be satisfied: current minting policy (`web3/contracts/nft-minting-policy.hl`) is a single-signature mint-only policy; CIP-25 metadata is immutable once minted
- On-chain revocation requires either (a) student cooperation to burn/send back the token, or (b) a new spending policy contract
- Admin notification explicitly instructs manual on-chain action if required
- OQ-06 remains open — contract upgrade needed to fully satisfy SC-E02-05

### F-16.2: On-chain revocation fails — PARTIAL

- Platform-level revocation always succeeds (DB transaction)
- No automated retry queue for on-chain update (infeasible; noted in admin notification)
- Admin is alerted in-app and must take manual action

### F-16.3: Token reward flagged for clawback — DONE

- `token_reward_status` ENUM extended with `'clawback_flagged'` (migration `2026_02_28_000003`)
- `RewardInvalidationService` sets `token_reward_status = 'clawback_flagged'` on refund
- Admin notified in-app; on-chain clawback not performed (per spec assumption A-07)

### F-16.4: Refund with no rewards delivered — DONE

- `RewardInvalidationService` checks `certificate_status` and `token_reward_status`; if neither is `minted`/`minting`, returns early with no action (idempotent no-op)

### F-16.5: Refund during in-progress airdrop — PARTIAL

- `'minting'` status treated as in-progress; `clawback_flagged` is set as best-effort
- No active exclusion mechanism to interrupt an in-flight airdrop transaction

## Remaining Cross-Cutting Gaps

1. **SC-E02-05 / On-chain revocation** — Current NFT minting policy does not support metadata updates or burns. Full F-16.1 on-chain revocation requires a contract upgrade. OQ-06 must be resolved with the smart contract team before this can be addressed.

2. **F-18.4 student ADA balance check** — The architectural decision is that the platform wallet funds minting fees (not the student). If the spec intent is that students pay ~1 ADA from their own wallet, this requires both a spec clarification and a client-side CIP-30 `getBalance()` check before submission.

3. **OQ-04 token reward bounds** — No platform-defined min/max for `token_reward_amount`. Only `> 0` is validated.

4. **F-05 certificate image — file upload** — Certificate image is a URL text input only. S3 file upload integration is deferred.

5. **OQ-07 single-tx airdrop** — Whether NFT certificate and token reward are delivered in a single on-chain transaction or two separate transactions is not verified. Current implementation calls separate scripts.

6. **F-19.1 email notification** — In-app notification on completion is implemented. Email notification remains out of scope.
