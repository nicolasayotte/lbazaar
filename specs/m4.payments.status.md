## F-01: Class Pricing in ADA | MUST — Done (~95%)

### F-01.1: ADA price display — DONE

**Backend:**
- `ExchangeRateService.getAdaJpyRate()` fetches ADA/JPY from CoinGecko (`app/Services/API/ExchangeRateService.php:17-62`)
- `ExchangeRateService.jpyToAda()` converts JPY to ADA (`ExchangeRateService.php:85-94`)
- `ExchangeRateService.addPriceInAdaToCourses()` decorates course collections (`ExchangeRateService.php:103-128`)
- Caching: 600s TTL for API rate, 60s TTL for fallback DB rate (`config/services.php:34-39`)

**API:**
- `GET /api/courses/{course}/ada-price` — public endpoint (`routes/api.php:77-78`, `app/Http/Controllers/API/CourseController.php:15-22`)

**Portal controller:**
- `CourseController::index()` adds exchange rate to search results (`CourseController.php:94`)
- `CourseController::details()` adds exchange rate with try/catch fallback (`CourseController.php:128-135`)

**Frontend:**
- Listing: `Price()` component in `resources/js/components/cards/Course.jsx:34-44` renders `"¥5,000 (~₳100.00)"` via `formatDualPrice()`
- Detail: `ClassInformation()` in `resources/js/pages/Portal/Course/Details.jsx:572-574` displays dual price

**Tests:**
- `tests/Feature/CoursePricingTest.php` — dual pricing assertions
- `tests/Unit/Services/API/ExchangeRateServiceTest.php` — 23 unit tests covering conversion logic
- `tests/Integration/CoinGeckoRateTest.php` — real API validation

### F-01.2: ADA price freshness (60s refresh, 5% drift) — DONE

- 60-second polling: `Details.jsx:67-87` — `setInterval(poll, 60_000)` calls `/api/courses/{course}/ada-price`
- Drift detection: `Details.jsx:76-78` — `Math.abs(new - old) / old > 0.05` triggers `driftWarning` state
- Initial price captured in `adaPriceAtLoad` ref (`Details.jsx:54`)
- Warning UI: `Details.jsx:696-700` — yellow caption: "ADA price has shifted >5% since page load"

**Gap:** Frontend-only; no backend test for polling/drift (acceptable — this is client behavior).

### F-01.3: ADA conversion unavailable — DONE

- `ExchangeRateService.isAvailable()` checks cache then live fetch (`ExchangeRateService.php:69-80`)
- Controller catches `\Throwable`, sets `ada_available = false` (`CourseController.php:128-135`)
- Frontend: ADA button disabled when `!adaAvailable` (`Details.jsx:682`)
- Unavailability notice: `Details.jsx:701-705` — "ADA price temporarily unavailable. You can still pay by credit card."
- Listing fallback: `Course.jsx:37-39` — shows JPY-only with `(ADA Unavailable)` label
- Stripe button uses independent `stripeAvailable` flag — unaffected

**Tests:**
- `ExchangeRateServiceTest.php` — `test_is_available_returns_false_when_both_fail()`
- `ExchangeRateServiceTest.php` — `test_adds_price_in_ada_gracefully_when_no_setting_exists()`

### F-01.4: Free classes (0 JPY) — DONE (display only)

- `ExchangeRateService.php:120` returns `null` for zero/negative prices
- `Details.jsx:569-577` — "Free" course type shows `translatables.texts.free` instead of price
- `Course.jsx:34-44` — `Price()` only renders dual price for "General" type; Free type renders empty

**Gap:** Spec says "Student can access the class directly" — direct access without payment flow not verified in enrollment logic. This depends on existing platform behavior for the "Free" course type, not new M4 code.

## F-02: Purchase Class with ADA | MUST — Done (~85%)

### F-02.1: Successful ADA purchase — DONE

- `CoursePurchaseService::buildPurchaseTransaction()` generates firm quote cached in Redis with configurable TTL (default 5 min)
- `CoursePurchaseService::submitPurchaseTransaction()` validates quote freshness and checks for duplicates via `CourseHistory.where('payment_status', ['pending', 'confirmed'])`
- Duplicate prevention uses `lockForUpdate()` (`CoursePurchaseService.php:184`) for race condition safety
- Frontend: `Details.jsx:48-105` implements countdown timer for quote expiry, blocks submission after window closes
- Pending state UI at `Details.jsx:47-72` with `LinearProgress` and blockchain explorer link

### F-02.2: ADA transaction confirmed (10 blocks) — DONE

- Hard-coded 10 confirmations in `config/services.php:43`
- `CoursePaymentController::handlePurchaseWebhook()` checks confirmations via `CoursePurchaseService::getTxConfirmations()`
- Access granted only after `pending` → `confirmed` transition (`CoursePurchaseService.php:264-266`)

### F-02.3: ADA transaction rejected — PARTIAL

- `CoursePurchaseService::failPurchaseTransaction()` marks `CourseHistory` and `WalletTransactionHistory` as 'failed'
- Idempotent design (returns `success=false` if already failed)

**Gaps:**
- **No on-chain failure detection loop.** Spec states "no timeout — system monitors until on-chain success or failure." Implementation relies solely on Blockfrost webhook. If a tx is dropped before reaching any block, it stays "pending" indefinitely.
- Frontend does not re-enable "Buy with ADA" button after failure without page reload.

### F-02.4: Insufficient ADA — DONE

- Node.js `build-purchase-tx.mjs` returns `status:501` for insufficient funds (caught at `CoursePurchaseService.php:90`)
- Frontend shows error and scrolls to credit card alternative (`Details.jsx:136-138`)

**Gap:** Boundary validation at `MIN_ADA=2000000` lovelace not verified in PHP layer (handled in Node.js).

### F-02.5: No wallet connected — DONE

- ADA button disabled when `!walletAPI` (`Details.jsx:642`)
- `WalletConnector` component prompts connection (`Details.jsx:717`)

### F-02.6: Student already owns class — DONE

- Duplicate check blocks purchase for students with existing `confirmed` or `pending` payment

### F-02.7: Concurrent purchase attempt — DONE

- `lockForUpdate()` prevents race conditions on concurrent submissions

### F-02.8: Quote window expiry — DONE

- Frontend countdown timer; backend validates quote TTL on submission; expired quotes rejected with re-initiation required

### Unimplemented Sub-features

- **Progressive confirmation count (X/10):** Spec requires showing "progressive confirmation count" during pending state. Frontend does not poll confirmation count — shows only "pending" with explorer link.

## F-03: Purchase Class with Stripe | MUST — Done (~95%)

### F-03.1: Successful credit card purchase — DONE

- `StripeService::createPaymentIntent()` (`app/Services/API/StripeService.php:55-161`) creates PaymentIntent with idempotency key (5-min window)
- Endpoint: `POST /api/stripe/payment-intent/{course}` (`routes/api.php:68`)
- `StripeService::handlePaymentSuccess()` (`StripeService.php:234-309`) creates `CourseHistory` enrollment on `payment_intent.succeeded`

### F-03.2: Card declined — DONE

- `StripeService::handlePaymentFailure()` (`StripeService.php:317-376`) handles `payment_intent.payment_failed`
- Updates `StripePayment` status to 'failed'; stores failure reason; no enrollment created

### F-03.3: Checkout abandoned — DONE

- `StripeService::handlePaymentCanceled()` (`StripeService.php:384-428`) handles `payment_intent.canceled`
- Sets status to 'canceled' only if still 'pending' (idempotent); no enrollment created

### F-03.4: Stripe service unavailable — DONE

- `StripeService::isAvailable()` (`StripeService.php:25-28`) checks for configured secret key
- `stripe_available` prop passed to frontend (`CourseController.php:154`)
- Credit card button disabled when `stripeAvailable === false` with explanation text

### F-03.5: Student already owns class — DONE

- Duplicate check in `StripeService::createPaymentIntent()` prevents creating checkout for already-purchased class

### Gaps

- **No 3D Secure intermediate state:** `requires_action` status not modeled
- **No chargeback webhook:** Only handles `succeeded/failed/canceled` — no `charge.dispute.created`

## F-12: Wallet Flow Verification | MUST — Partial (~65%)

### F-12.1: External wallet connects — DONE

- `WalletConnector.jsx:150-251` — full connection flow for Eternl, Flint, Nami
- `enableWallet()` retrieves wallet API; `getWalletInfo()` calls backend `/wallet/info`
- Address, balance, stake key displayed after connection
- Backend: `Web3WalletController::info()` calls Node.js `wallet-info.mjs`

### F-12.2: Unsupported wallet — PARTIAL

- `checkIfWalletFound()` (`WalletConnector.jsx:156-162`) detects if wallet extension is installed
- Generic error dispatched: `translatables.wallet_error.not_found`

**Gap:** No list of supported wallets shown to user. No distinction between "not installed" and "not supported."

### F-12.3: Wallet disconnects mid-session — NOT IMPLEMENTED

- No event listeners for `onAccountsChanged`, `onNetworkChanged`, or connection drop
- No heartbeat/polling to detect stale connections
- Once `walletAPI` is set, it is never cleared on disconnect
- Manual switch only via `handleWalletSwitch()` (`WalletConnector.jsx:409-418`)

### F-12.4: Wrong network detection — DONE

- `checkNetwork()` (`WalletConnector.jsx:205-217`) validates wallet network ID against `config('services.cardano.network_id')`
- Network ID passed from backend via `HandleInertiaRequests.php:51`
- Connection blocked with clear Mainnet/Testnet error message
- State cleaned up on mismatch

### F-12.5: Wallet disconnect during pending tx — PARTIAL

- Backend (Blockfrost webhooks) continues monitoring on-chain regardless of client wallet state
- `pendingPayment` tracked in Inertia props — survives page reload

**Gap:** No disconnect detection (depends on F-12.3). No UX indicating "your wallet disconnected but your pending transaction is still being tracked."

## Cross-Cutting Gaps

1. **On-chain tx failure detection:** No fallback if Blockfrost webhook is lost or delayed. Pending payments can be orphaned.
2. **Progressive confirmation display (X/10):** Spec requires it; not implemented in any view.
3. **Admin refund UI:** `StripeService::refund()` exists but no admin page or API endpoint exposes it.
4. **Wallet disconnect resilience:** No detection, no graceful degradation, no reconnect prompt.
5. **Both-paths-down combined message (F-04.3):** Individual disabled messages shown but no unified "purchases temporarily unavailable" notice.
