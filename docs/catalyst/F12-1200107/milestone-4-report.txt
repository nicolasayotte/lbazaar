ROUND 2 — ITEMS ADDRESSED

Walkthrough video: A detailed end-to-end walkthrough video demonstrating teacher and
student journeys will be prepared and delivered as part of the final milestone.

1a. JPY→ADA conversion mechanism and ADA opt-out for teachers.

    The conversion mechanism is now fully documented. Teachers set a class price in JPY
    (the single source-of-truth). ADA is derived for display and at checkout via the
    CoinGecko API (cached 10 min, with a DB-setting fallback rate). At checkout, the ADA
    amount is locked for a 5-minute quote window; the course page live-polls the rate and
    shows a drift warning if it moves more than 5%, prompting the student to re-confirm.
    Prices are shown dually everywhere (e.g. ¥10,000 (~₳100)). Full detail is in
    docs/integrations.md (§9 CoinGecko Exchange Rate API) and docs/data-flows.md
    (§ ADA payment flow). Every paid class already exposes both a "Buy with ADA" and a
    "Pay with Credit Card" path, so students of crypto-averse teachers can pay entirely
    by card today. A per-class payment-method toggle (card only / ADA only / both) is
    added to the roadmap for a future milestone.

1b. "Edit Class" changes not reflected after save — fixed and deployed.

    Root-caused as a validation mismatch: the course form sends a plural `categories`
    array (introduced with multi-category support) but the server still required the
    legacy singular `category` field. On create the singular value was present in the
    application data and slipped through; on edit the course object has no singular
    `category`, validation failed silently, and the form appeared to do nothing. The
    automated tests gave false confidence because they submitted a hand-built payload
    containing the singular field that the real UI no longer sends. Fix: aligned
    CourseRequest validation to the plural `categories` field, reconciled the create
    path, and replaced the misleading tests with ones that submit the real frontend
    payload shape. Fix is deployed to staging. (app/Http/Requests/CourseRequest.php,
    app/Repositories/CourseRepository.php, tests/Feature/.../CourseRewardConfigTest.php)

    Related clarification: the price field on the edit form is intentionally read-only
    — a class's price is set in the class application and fixed once the community
    approves it (the community votes on a class as priced). The field previously gave no
    indication of this; it now displays an explanatory note in English and Japanese.

1c. One video / one lesson per course — multi-lesson is supported via Course Packages.

    The platform is not limited to a single lesson. A single Course record is one unit
    of content, but teachers compose multi-lesson curricula via Course Packages: a
    teacher creates a package and groups multiple courses (lessons) under it
    (CoursePackage → many Course, junction table course_package_courses). On a course's
    detail and attendance pages, students see all sibling lessons in the same package and
    a "complete the classes to earn the badge" progression. A live Course also supports
    many CourseSchedules, making recurring/multi-session classes a first-class concept.
    There is not yet a dedicated "browse packages" page or a bundled package checkout —
    those gaps are acknowledged and on the roadmap. The walkthrough video for the final
    milestone will demonstrate building and consuming a multi-lesson package explicitly.

1d. Teacher payout configuration — payouts are automatic; visibility UI is the gap.

    There is no manual wallet-configuration screen because payouts require none: each
    user is automatically assigned a custodial Cardano wallet derived deterministically
    from their user ID (get-custodial-address.mjs). When a student buys with ADA, a
    single on-chain transaction splits the payment — the teacher's custodial address
    receives their share and the platform receives its commission — with no intermediary
    holding funds. The split is governed by an admin commission setting (default 20%
    platform / 80% teacher). For credit-card (Stripe) sales, the platform collects the
    full card payment and teacher settlement is handled off-platform; per-teacher
    automated card payout (e.g. Stripe Connect) is not yet implemented. What is
    genuinely absent is teacher-facing visibility: there is no earnings or balance
    dashboard yet. A teacher earnings/payout view is added to the roadmap. The payout
    and commission mechanics are documented in docs/data-flows.md.

2.  Economic model, token utility, and revenue model.

    Token reward: the mechanics are fully built and tested (per-course opt-in fungible
    mint on completion, metadata, invalidation on refund — see docs/token-reward.md).
    The economic model is deliberately deferred: defining monetary utility for the token
    in Japan raises real regulatory questions (classification, obligations). Rather than
    ship a tokenomics design that may need to be unwound, the model is held open until
    the regulatory path is clear. The token currently functions as a completion-reward
    primitive with no promised monetary value. docs/token-reward.md states the
    implemented mechanics and the explicitly open design questions.

    Revenue model: the platform takes a commission on sales (default 20% platform / 80%
    teacher), applied on-chain for ADA payments and via the platform account for card
    payments. This is documented in docs/data-flows.md.

3.  Discord community governance — scope, direction, and evidence.

    The governance layer is implemented and the Discord bot is operational. The two
    decisions currently governed by the community are: (a) who becomes a teacher — a
    prospective teacher's application creates a vote in the community channel, and
    passing the vote grants teacher status; and (b) which classes get published — a
    teacher applies per class through the same voting mechanism
    (CourseApplication.approved_at) before a class can go live. Both are enforced in
    code; the "your information needs to be verified" state a reviewer sees is a pending
    community vote, not a broken flow. The vote-passing threshold is itself a governance
    parameter (vote-passing-percentage). Future candidate decisions include featured /
    curated content and categories, and dispute / quality signals on courses. For the
    final milestone we will provide: the Discord server invite, current member count, an
    activity snapshot, examples of real teacher/class governance votes, and feedback
    from partners and interview participants on the governance model.

4.  Test classes — local-government-related organization and school.

    Both acceptance criteria were met during this milestone and are evidenced in the
    Marketing & Sales report (Google Doc linked in the PoA, §"Sales, Marketing &
    Hearings", dated May 24, 2026).

    Additional school(s) holding Web3 / SDG-related test classes: more than 10 test

    with ongoing test-class use — and Seisen University (Prof. Kanekiyo, President
    Yamamoto), where the platform was introduced and demonstrated with discussions toward

    Additional local-government-related organization: Akariyu Suzuran (灯湯すずらん),
    Chino City — a Lake Shirakaba-area community facility supported by Chino City as
    part of regional-revitalization initiatives, prepared as a regional implementation
    site (facility opens June 2026; detail to be reported at close-out). IBM Regional DX
    Center Nagano — established under a tripartite agreement among IBM Japan, Nagano
    Prefecture, and Nagano City ("Smart City NAGANO"); attended and discussed platform

Fixed (code changes):

1. Reviewer concern: Wallet connection supports only 3 wallets (Nami deprecated).
   Resolution: The connector now dynamically detects every installed CIP-30 wallet from window.cardano instead of a hardcoded list — no wallet is pinned, and newer wallets appear automatically. (resources/js/components/cards/WalletConnector.jsx)

2. Reviewer concern: Course language filter shows only "All"; no language field at creation.
   Resolution: Added a required language selector to the course create/edit form and a canonical config/languages.php; the browse-page filter is now populated from it.

3. Reviewer concern: "Cancel Booking" button appears non-functional.
   Resolution: For paid (ADA/Stripe) bookings the button now shows a clear admin-refund notice instead of silently doing nothing; free/points bookings keep the working self-cancel. Refunds remain admin-handled by design (the refund path the reviewer saw passing in the logs).

4. Reviewer concern: 7 skipped Playwright tests.
   Resolution: All seven were conditional skips guarding missing fixtures; each was diagnosed and fixed. The suite is now 225 passed / 0 skipped / 0 failed / 0 flaky (was 208 passed / 1 flaky / 7 skipped). While fixing them we also found and fixed a real bug — the admin General Settings form could not be saved on a fresh database because two required settings were never seeded.

5. Reviewer concern: No CI for the testing pipeline.
   Resolution: Added a GitHub Actions workflow (.github/workflows/ci.yml) running the JS, web3, and PHPUnit suites on every push and pull request.

Clarified (already implemented / by-design):

1. Reviewer concern: Teacher account stays locked / no verification email.
   Clarification: By design. Teacher onboarding runs through a Discord-bot community vote — a person is voted in as a teacher, then applies per-class through the same vote (CourseApplication.approved_at). The "your information needs to be verified" message reflects that pending community vote, not a broken flow. A pre-approved demo teacher account is seeded so the reviewer can test the full teacher workflow immediately - testing credentials (temporary)

   email: demo-teacher@lebazaar.com
   password: Demo1234!

2. Reviewer concern: Certificate NFT looks like CIP-25, not CIP-68.
   Clarification: Correct — the certificate is CIP-25 by design (metadata under label 721). The (100)/(222) asset-name prefixes follow CIP-68's labelling convention for clean reference/user-token separation, but metadata is not stored in a reference-token datum (the defining feature of CIP-68).
   Purpose of the paired token: the two tokens are a revocation scheme. Each certificate mint produces a (222) user token, which is airdropped to the student as the certificate they hold, and a (100) reference token, which is retained in the platform wallet. To revoke a certificate, the platform burns its (100) reference token and writes a CIP-25 metadata update (revoked: true, with timestamp) under label 721. The student's (222) token remains in their wallet, but the on-chain reference token and updated metadata record the certificate as revoked. The pairing therefore gives the platform a burnable, on-chain handle to mark revocation without requiring the holder's signature. Documented in docs/certificate-minting-api.md.

3. Reviewer concern: Commenting on courses/lessons not found.
   Clarification: Implemented as the Course Feedback feature (CourseFeedback model, Feedback.jsx, CourseFeedback.jsx) it basically is a simple extension that allows commenting on each lessons instead of only on the whole course (as before).

4. Reviewer concern: Credit-card buyers viewing their certificate NFT.
   Clarification: Works today — certificate airdrop is independent of payment method. Each user as a custodial wallet by default if they do not link a non-custodial wallet: this is where the certificates get sent.

Documented / planned:

1. Reviewer concern: Token reward purpose & tokenomics unclear.
   Outcome: Documented the current mechanics and the explicitly open design questions (utility vs. loyalty, supply/caps, use cases) in docs/token-reward.md, rather than overstate a finalized economic model.

2. Reviewer concern: Separate teacher/student account types.
   Outcome: Acknowledged as input; the split exists today because teachers carry community-approval state and payout/commission settings. Teacher accounts have numerous extra features compared to student accounts.

3. Reviewer concern: Draft content, category reorganization, example/intro lessons.
